# Milestone 3 — XDG-Compliant Data Storage

**Status**: complete

## Scope

Replace the monorepo-dependent `_find_repo_root()` path resolution with XDG-compliant home directory storage. After this milestone, doc-hub can run as a standalone pip-installed package without any UV workspace present.

Resolution chain for the data root:
1. `DOC_HUB_DATA_DIR` environment variable (explicit override)
2. `$XDG_DATA_HOME/doc-hub` if `XDG_DATA_HOME` is set
3. `~/.local/share/doc-hub` (XDG default)

The plugins directory (used by Milestone 2) lives at `{data_root}/plugins/`.

## Files to Create/Modify

### Modify: `src/doc_hub/paths.py`

**Remove entirely:**
- `_find_repo_root()` function — this is the monorepo coupling that must go.

**Rewrite `data_root()`:**

```python
def data_root() -> Path:
    """Return the root data directory.

    Resolution order:
    1. DOC_HUB_DATA_DIR env var (explicit override)
    2. XDG_DATA_HOME/doc-hub (if XDG_DATA_HOME is set)
    3. ~/.local/share/doc-hub (XDG default)

    The directory is NOT created automatically — callers that write
    files are responsible for calling mkdir(parents=True, exist_ok=True).
    """
    env_override = os.environ.get("DOC_HUB_DATA_DIR")
    if env_override:
        return Path(env_override).expanduser().resolve()

    xdg_data = os.environ.get("XDG_DATA_HOME")
    if xdg_data:
        return Path(xdg_data).expanduser().resolve() / "doc-hub"

    return Path.home() / ".local" / "share" / "doc-hub"
```

**Keep unchanged:**
- `corpus_dir(corpus)` → `data_root() / corpus.slug`
- `raw_dir(corpus)` → `corpus_dir(corpus) / "raw"`
- `chunks_dir(corpus)` → `corpus_dir(corpus) / "chunks"`
- `manifest_path(corpus)` → `raw_dir(corpus) / "manifest.json"`
- `embedded_chunks_path(corpus)` → `chunks_dir(corpus) / "embedded_chunks.jsonl"`
- `embeddings_cache_path(corpus)` → `chunks_dir(corpus) / "embeddings_cache.jsonl"`

**Add new helper:**

```python
def plugins_dir() -> Path:
    """Return the local plugins directory: {data_root}/plugins/.

    Used by the plugin discovery system (doc_hub.discovery) to scan
    for local .py plugin files.
    """
    return data_root() / "plugins"
```

**Refactor corpus helpers to accept `slug: str` instead of `Corpus`:**

Currently `corpus_dir()`, `raw_dir()`, `chunks_dir()` etc. all take a `Corpus` object and access `.slug`. To decouple from the `Corpus` model (which is being refactored in Milestone 5), these should accept either a `Corpus` or a plain `str` slug. The simplest approach:

```python
from __future__ import annotations

import os
from pathlib import Path
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from doc_hub.models import Corpus


def corpus_dir(corpus_or_slug: Corpus | str) -> Path:
    """Per-corpus root directory: {data_root}/{slug}/."""
    slug = corpus_or_slug if isinstance(corpus_or_slug, str) else corpus_or_slug.slug
    if not slug or "/" in slug or "\\" in slug or slug.startswith("."):
        raise ValueError(
            f"Invalid corpus slug: {slug!r}. Slugs must be non-empty, "
            "must not contain path separators, and must not start with '.'"
        )
    return data_root() / slug
```

**Slug validation:** Since the slug becomes a filesystem directory name AND a database primary key, validate it at the boundary where it enters the path system. This prevents path traversal (e.g., `slug="../etc"`) and empty slugs that would resolve to the data root itself. Dotfile slugs (starting with `.`) are also rejected to prevent hidden directories.

**Important:** The `from doc_hub.models import Corpus` import must move inside a `TYPE_CHECKING` guard. Currently `paths.py` imports `Corpus` at the top level (line 25 of current `paths.py`: `from doc_hub.models import Corpus`). With the guard, the import is only used for type checking (mypy/pyright), not at runtime. The `isinstance(corpus_or_slug, str)` check at runtime doesn't need the `Corpus` type — if it's not a string, it's assumed to have a `.slug` attribute, accessed via `corpus_or_slug.slug`.

Apply the same `Corpus | str` pattern to ALL path helpers: `raw_dir`, `chunks_dir`, `manifest_path`, `embedded_chunks_path`, `embeddings_cache_path`. Each uses the same `slug = corpus_or_slug if isinstance(corpus_or_slug, str) else corpus_or_slug.slug` extraction. This maintains backward compatibility with existing code that passes `Corpus` objects while enabling new code (Milestones 6, 7) to pass plain slug strings.

**Runtime behavior:** With `from __future__ import annotations` (already present), the `Corpus | str` annotation is a string at runtime, so the `Corpus` type is never actually loaded. The `isinstance(corpus_or_slug, str)` check works because `str` is a builtin.

### Modify: `src/doc_hub/eval.py`

Update `_eval_dir()` to follow a similar XDG pattern:

def _eval_dir() -> Path:
    """Return the eval directory.

    Resolution order:
    1. DOC_HUB_EVAL_DIR env var
    2. {data_root}/eval/

(The current fallback `Path.cwd() / "eval"` is fragile and monorepo-specific. Falling back to `{data_root}/eval/` is more predictable.)

**Consistency with `data_root()`:** Apply `expanduser().resolve()` to the env var path for consistency with `data_root()`. Both functions handle user-provided paths the same way.

## Design Decisions

1. **No auto-creation of data_root**: The function returns the path but does not `mkdir`. Callers that need to write (fetch, parse, embed stages) call `mkdir(parents=True, exist_ok=True)`. This keeps the path function side-effect-free and testable.

2. **`expanduser()` and `resolve()`**: Applied to env var paths to handle `~` and relative paths correctly.

3. **Backward-compatible function signatures**: Accepting `Corpus | str` means existing code doesn't break while new code can use plain strings.
