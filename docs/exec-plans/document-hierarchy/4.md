## Detailed Specification

### 1. New function: `_parse_sections(llms_txt_content, url_pattern) -> list[dict]`

Parse markdown headings from the llms.txt content and track which extracted URLs fall under each heading.

```python
def _parse_sections(llms_txt_content: str, url_pattern: str) -> list[dict]:
    """Parse section headings from llms.txt and map URLs to sections.

    Walks the llms.txt line by line. When a markdown heading is encountered
    (## Heading, ### Heading, etc.), start a new section. URLs matching
    url_pattern found between this heading and the next are assigned to
    this section.

    Args:
        llms_txt_content: Raw text of the llms.txt file.
        url_pattern: Regex pattern for extracting doc URLs (same as used
            by the fetcher for URL extraction).

    Returns:
        List of section dicts:
        [
            {
                "title": "API Reference",
                "heading_level": 2,
                "urls": ["https://ai.pydantic.dev/api/base.md", ...]
            },
            ...
        ]

        URLs that appear before any heading are assigned to a synthetic
        section with title="" and heading_level=0.
    """
```

Required behavior:
- Preserve the order of sections as they appear in the file.
- Treat `#` as the document title, not a section bucket; start structural sections at `##` and below.
- Assign URLs to the most recent heading above them, not merely to the next heading level.
- Deduplicate URLs within each section while preserving first-seen order.
- Preserve headings even when a heading has no URLs so the manifest can still carry structural hints for browsing.
- If the llms.txt contains URLs before the first `##`/`###` section heading, place them in a synthetic root section with `title=""` and `heading_level=0`.

### 2. Update `write_manifest()` signature

Add optional `sections` parameter. The existing function is at `llms_txt.py:102-121`. Modify it by:
1. Adding `sections: list[dict] | None = None` parameter
2. Conditionally adding `"sections"` to the manifest dict

The existing function body builds the manifest dict and writes it. Insert the sections handling after the manifest dict is built (after the `"files"` key) and before writing to disk:

```python
def write_manifest(
    results: list[DownloadResult],
    output_dir: Path,
    sections: list[dict] | None = None,
) -> None:
    """Write a JSON manifest of download results to ``output_dir/manifest.json``."""
    manifest = {
        "total": len(results),
        "success": sum(1 for r in results if r.success),
        "failed": sum(1 for r in results if not r.success),
        "files": [
            {
                "url": r.url,
                "filename": r.filename,
                "success": r.success,
                "error": r.error,
                "content_hash": r.content_hash,
            }
            for r in sorted(results, key=lambda r: r.filename)
        ],
    }
    if sections is not None:
        manifest["sections"] = sections

    manifest_path = output_dir / "manifest.json"
    manifest_path.write_text(json.dumps(manifest, indent=2))
    log.info("Manifest written to %s", manifest_path)
```

**Backward compatibility**: The `sections` parameter defaults to `None`. When `None`, the `"sections"` key is NOT added to the manifest. This means:
- Existing callers that don't pass `sections` are unaffected
- `load_manifest()` only reads the `"files"` key and ignores everything else, so even if `"sections"` is present, it won't break anything
- `build_document_tree()` reads `manifest.json` directly and uses `data.get("sections")`, which returns `None` if the key is absent
